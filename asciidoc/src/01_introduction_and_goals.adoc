== Introduction and Goals

Die Obelix-Applikation ist eine moderne, verteilte E-Commerce-Plattform für den Verkauf und die Verwaltung von Hinkelsteinen. Das System wurde als Klassenprojekt entwickelt und demonstriert den Einsatz von Microservice-Architektur, OpenAPI-Standards und modernen DevOps-Praktiken.

**Fachliche Ziele:**

Das Obelix-System ermöglicht den digitalen Vertrieb von Hinkelsteinen und bietet eine zentrale Verwaltungsplattform für das Produktsortiment. Kunden können selbstständig Hinkelsteine durchsuchen und bestellen, während Administratoren das Angebot pflegen.

**Kernfunktionen:**

* *Customer Web UI:* Kundenzugang zum Hinkelstein-Katalog mit Such- und Bestellfunktionen
* *Admin Web UI:* Verwaltungsinterface für CRUD-Operationen auf Hinkelsteinen
* *RESTful API:* Öffentliche Schnittstellen für externe Entwickler zur Integration
* *Monitoring & Observability:* Vollständige Überwachung durch Zipkin, Prometheus und Grafana

**Technische Kernanforderungen:**

[options="header",cols="1,3"]
|===
|Kategorie |Anforderung

|*Architektur*
|Mindestens 3 unabhängige Microservices mit klarer Verantwortlichkeitstrennung

|*API-Spezifikation*
|OpenAPI 3.x (OAS) für alle Service-Schnittstellen, code-first generiert (z.B. springdoc-openapi)

|*API-Dokumentation*
|Alle OAS Properties mit Beschreibungen und Beispielen versehen

|*Frontend*
|Ausschließliche Nutzung generierter OpenAPI-Clients (z.B. openapi-generator)

|*Observability*
|Integration von Zipkin (Distributed Tracing), Prometheus (Metrics), Grafana (Dashboards)

|*Dokumentation*
|arc42-Architekturdokumentation in AsciiDoc mit PlantUML-Diagrammen

|*Integration Guide*
|Umfassende Anleitung für externe Entwickler mit auto-generierten Code-Samples aus Tests

|*CI/CD*
|Automatisierte Build-Pipeline und Deployment in Produktion (BBW MaaS) bei jedem main-Branch Commit

|*Code Quality*
|Alle Änderungen via Pull Requests mit mindestens 3 Reviews

|*Testing*
|Automatisierte Tests bei jedem PR
|===

**Nicht-funktionale Anforderungen:**

* *Keine Authentisierung:* Admin-UI bewusst ohne Session-Management (Projektanforderung: "easy")
* *LLM-Nutzung:* AI-generierter Code soll verborgen bleiben (keine typischen AI-Muster)
* *Kollaboration:* Arbeitsteilung durch die Klasse, gemeinsame Verantwortung für Endergebnis

.Form
Tabellarische Darstellung der Qualitätsziele mit möglichst konkreten Szenarien, geordnet nach Prioritäten.
****

[options="header",cols="1,2,2"]
|===
|Priorität |Qualitätsziel |Szenario/Operationalisierung

|1
|*API-First Entwicklung*
|Alle Service-Schnittstellen sind vollständig mit OpenAPI 3.x spezifiziert. Frontend-Entwickler können parallel zu Backend-Entwicklung arbeiten, indem sie generierte Clients nutzen. Externe Entwickler verstehen die API ohne zusätzliche Dokumentation.

|2
|*Wartbarkeit & Verständlichkeit*
|Neue Entwickler können sich innerhalb von 2 Stunden durch die arc42-Dokumentation in das System einarbeiten. Alle Architekturentscheidungen sind nachvollziehbar dokumentiert. Code-Samples in der Dokumentation sind immer aktuell (auto-generiert aus Tests).

|3
|*Observability*
|Bei jedem Request ist der vollständige Trace über alle Microservices in Zipkin sichtbar. Performance-Probleme können durch Prometheus-Metriken innerhalb von 5 Minuten identifiziert werden. Grafana-Dashboards zeigen Systemzustand in Echtzeit.

|4
|*Continuous Deployment*
|Jeder Merge in den main-Branch führt automatisch zu einem Deployment in Produktion. Build-Fehler werden innerhalb von 10 Minuten durch CI-Pipeline erkannt. Rollback ist durch Docker-Container-Versionierung jederzeit möglich.

|5
|*Teamfähigkeit*
|Mindestens 3 Reviews pro Pull Request stellen Code-Qualität sicher. Code-Ownership ist durch CODEOWNERS klar definiert. Merge-Konflikte werden durch Branch-Protection und klare Verantwortlichkeiten minimiert.
|===
